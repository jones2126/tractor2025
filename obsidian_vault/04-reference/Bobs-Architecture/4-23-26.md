**Tractor Autonomy Project — Summary (from known context, no guessing)**
 
**Goal**  
Build a fully autonomous self-driving lawn tractor capable of GPS-guided navigation with remote teleoperation fallback. The base platform is a **Cub Cadet EFI lawn tractor**. Longer-term intent is to eventually apply lessons to larger equipment (a John Deere 5055E for hay cutting was mentioned in earlier conversations).
 
**Hardware**  
**Compute**

- Raspberry Pi 5 (hostname: raspberrypi, user: al, IP 192.168.1.151)
- Teensy 4.1 microcontroller (connected via USB serial to RPi at 460800 baud, symlinked as /dev/teensy)

**Motor Controllers**

- IBT-2 H-bridge × 2 — one for steering, one for transmission (or a second actuator; the steering IBT-2 drives the steering motor via PWM pins 5 and 6 on the Teensy)
- Pololu JRK G2 21v3 — controls the transmission linear actuator via Serial3 on the Teensy at 9600 baud

**Steering**

- Ackermann/steering wheel style
- Steering position feedback via analog pot on Teensy pin A9 (range 0–1023)
- PID loop on Teensy running at 10 Hz; setpoint comes from RC radio (manual) or cmd_vel.angular_z (auto)

**Transmission**

- Linear actuator controlled by JRK G2
- 10-bucket system; JRK feedback range 336–3670
- Neutral = 2800 (approximately; transmissionNeutralPos in code is currently 2985 — **this may need field calibration**)
- Full reverse = 4095, full forward = 0

**Power** _(partial — gaps noted below)_

- Tractor 12V battery is the assumed main power source
- Voltage sense line is wired to Teensy analog pin 18 through a resistor divider; code constants are VOLTAGE_DIVIDER_RATIO = 5.0, ADC_REF_VOLTAGE = 3.3
- **Gap: Exact battery → RPi regulation chain (buck converter model/rating) not recorded**
- **Gap: Whether there is a common ground tie between the Teensy, IBT-2, and RPi is not confirmed in memory — this was the subject of yesterday's problem**

**Sensors**

- 2× ArduSimple F9P GPS (one for position/GGA via /dev/gps-base-link; one for heading/RELPOSNED via /dev/gps-heading)
- OAK-D stereo camera (DepthAI 2.30.0.0)
- NRF24L01 radio modules (Teensy receives; handheld transmitter unit is a separate Arduino/Teensy running rccntrl_nrf24_20251128.cpp)
- PCA9685 servo board (I2C) driving LED status tower

**Radio Control Unit** (handheld transmitter)

- Reads 5 analog pots (steering, throttle, transmission, pot4, voltage) and 3 digital buttons/switches
- Sends 14-byte packed struct at 10 Hz to tractor Teensy
- Receives 14-byte ACK payload back containing GPS status (0–3 enum) and button echoes
- 4× PL9823 NeoPixel LEDs for signal quality, e-stop, mode, and GPS status

**Software Stack**  
**OS:** Raspberry Pi OS 64-bit  
**Python services on RPi (all run as systemd services):**

|   |   |   |
|---|---|---|

**Navigation:** Pure pursuit (pure_pursuit.cpp) — ROS1 node consuming odometry, publishing cmd_vel (Twist); angular_z carries steering angle, linear_x carries speed  
**UDP port conventions:**

- 6002 = GPS/RTK state (rtcm_server → bridge, LED controller)
- 6003 = Teensy bridge broadcast status
- 6004 = cmd_vel commands (JSON {"linear_x": float, "angular_z": float})

**Networking:** ZeroTier (ztuze7ml6g), rover at 192.168.193.76; WiFi profiles cui_bono (home, priority 100) and Pixel_4952 hotspot (field, priority 50)
 
**Current Status**  
**What works:**

- NRF24 radio link between handheld unit and Teensy (10 Hz, ACK payload with GPS status echoed back)
- GPS RTK pipeline: base F9P → RTCM server → UDP 6002 broadcast with fix quality, heading, carrier
- Teensy serial bridge: parses Teensy output, broadcasts on UDP 6003, forwards GPS status to Teensy, accepts cmd_vel on UDP 6004
- LED status tower (PCA9685, I2C)
- Teleop console (WebRTC video + map dashboard)
- JRK G2 transmission control with 10-bucket RC pot system (calibration may need field tuning)
- ZeroTier SSH recovery path

**Known gaps / pain points:**

1. **Yesterday: Serial communication between Teensy and RPi was lost** — root cause in memory is that common ground between Teensy and JRK was missing (328Ω measured instead of 0Ω), but the serial loss to the RPi may involve a related grounding or voltage issue. The exact wiring of the Teensy USB/serial connection relative to the IBT-2 power ground is **not fully documented in memory**.
2. **A Teensy was blown up** — the exact failure mode isn't recorded in detail, but the concern is voltage/signal line routing between the battery, IBT-2, and Teensy.
3. **Voltage regulation chain** for the RPi is not confirmed (buck converter specs unknown).
4. **Transmission neutral position** (transmissionNeutralPos = 2985 in firmware vs. recalibrated range 336–3670 with neutral ~2800) — may need field verification.
5. **Visual odometry** (OAK-D fallback for GPS dropout) not yet implemented; vo_gps_arbiter.py pattern identified but not built.
6. **Carrier PCB** not yet fabricated (KiCad design in progress).
